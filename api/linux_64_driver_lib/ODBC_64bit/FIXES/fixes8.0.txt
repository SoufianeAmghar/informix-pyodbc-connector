     FIXES.TXT
     Progress(R) DataDirect(R)
     Progress DataDirect for ODBC Drivers
     Release 8.0
     December 2024


This FIXES.TXT file contains information for Progress(R) DataDirect  
for ODBC Drivers, which includes the following products:
* Progress DataDirect for ODBC Drivers (32- and 64-bit)
* Progress DataDirect Connect for ODBC
* Progress DataDirect Connect64 for ODBC
* Progress DataDirect Connect XE (Extended Edition) for ODBC
* Progress DataDirect Connect64 XE for ODBC

* You must have a component build number in the driver version string
that is equal to or higher than the one listed in this file to have the 
fix for the driver failure.

* You must have a component build number in the UNIX Driver Manager (DM) 
version string that is equal to or higher than the ones listed in this 
file to have the fix for the Driver Manager failure. The Driver Manager 
files containing the version string are as follows:

  On AIX: odbc.so
  On HP-UX PA-RISC: libodbc.sl
  On Solaris, Linux, and HP-UX Itanium: libodbc.so

* You must have a component build number in the ICU version string that 
is equal to or higher than the one listed in this file to have the fix 
for the ICU failure. On Windows, this file is xxICUnn.DLL, and on UNIX, 
libxxicunn.yy, where  is iv for the 32-bit library and dd for the 
64-bit library, nn represents the driver level number, and yy represents 
the appropriate file name extension on UNIX.

All drivers, except the flat-file and Salesforce drivers have a 
version string of the format:

XX.YY.ZZZZ(bAAAA, UBBBB)
or
XX.YY.ZZZZ(BAAAA, UBBBB)

All flat-file drivers have a version string of the format:

XX.YY.ZZZZ(bAAAA, uBBBB, FCCCC)

The Salesforce driver has a version string of the format:

XX.YY.ZZZZ(bAAAA, UBBBB, SDDDDDD)

The MongoDB and Oracle Service Cloud drivers have a version string of the format:

XX.YY.ZZZZ(bAAAA, UBBBB, JDDDDDD)

The Driver Manager on UNIX and Linux has a version string of the format:

XX.YY.ZZZZ(UBBBB)

The component for the Unicode conversion tables (ICU) has a version 
string of the format:

XX.YY.ZZZZ

The Schema Definition Tool has a version string of the format:

EEEEEE

where:

XX     is the major version of the product 
YY     is the minor version of the product
ZZZZ   is the build number of the driver or DM component 
AAAA   is the build number of the driver's bas component 
BBBB   is the build number of the driver's or DM's utl component 
CCCC   is the build number of a flat-file driver's flt component
DDDDDD is the build number of the Salesforce, MongoDB or OSC driver's jar component
EEEEEE is the build number of the Schema Definition Tool (schematool.jar)

For example for a flat-file driver, 8.01.0012 (b0001, u0002, F0001)
                                          |__|  |___|  |___|  |___|
                                         Driver  Bas    Utl    Flt

This fixes.txt file lists the build number of the component(s) 
that changed to resolve the failure. For example, if a driver's bas 
component changed, you may see b0001. 

The fixes.txt file lists both the DM component number and the utl 
component number for a DM failure.

On Windows XP and 2003, you can check the driver version string through 
the properties of the driver DLL. Right-click the driver DLL and select 
Properties. The Properties dialog box appears. On the Version tab, 
click File Version in the "Other version information" list box.

On all supported Windows platforms, you can also check the version
string of the drivers from the ODBC Administrator, on the About tab.

On UNIX, you can check the version string by using the ivtestlib or 
ddtestlib utility shipped with the product. This utility is located in 
<install_directory>/bin. Use the following command line:

ivtestlib <shared_object>

For example, for the Spark SQL Wire Protocol driver on Solaris:

>ivtestlib ivspark28.so
>
>08.01.0001 (B0002, U0001)

Note that the Spark SQL Wire Protocol driver is not a flat-file driver; 
therefore, there is no flt component listed in the example.

For example, for the Driver Manager on Solaris:

>ivtestlib libodbc.so
>
>08.01.0001 (U0001)

For example, for ICU on Solaris:

>ivtestlib libivicu28.so
>
>08.01.0001

NOTE: On Solaris, AIX, and Linux, the full path to the driver does not 
have to be specified for ivtestlib and ddtestlib. The HP-UX version of 
ivtestlib and ddtestlib, however, requires the full path.
